Folks,

As you may have seen if you've looked at your developer console or browsed the documentation on the developer site today, we are changing the way authentication and authorization work in the Platform API. I wanted to explain why we're making this change and how it affects you, as developers.

First, let's talk about what's changing. The whole process is documented on our developer site here:

https://www.sugarsync.com/dev/get-auth-token-example.html

The short version is that we've updated the authorization protocol so that apps no longer need to store users' passwords in order to acquire access tokens that can be used to access their accounts. Instead, apps use a password once to acquire a refresh token. This refresh token can then be used to get an access token, and it's persistent, so it can be saved and used again later.

The most important benefit of this change is that there is now no reason for an app to store a user's SugarSync password: you can simply store the refresh token, instead. A secondary benefit is that users can see which apps they have allowed to access their accounts, and they can revoke this access on an app-by-app basis via the SugarSync Account page.

Of course, we obviously can't break the many apps users depend on already, so we are transitioning to this new model gradually. As of today, all new developers must use the new authorization model, but existing developers can continue to use the old approach. We will continue to support the old mechanism for at least six months from today, and after that point will make a decision regarding how to decommission it. After November 23, the old interface may stop working with little or no notice.

We believe the new authorization framework is much more secure and provides a better user experience than the old interface, so we strongly urge you to adopt it sooner rather than later. If you have any questions or concerns about this change, please send a message to the mailing list or, if you'd rather, email me directly.

Thanks for supporting SugarSync and stay tuned—there are more exciting changes coming down the pipe.

Jim Wong (jwong@sugarsync.com)
